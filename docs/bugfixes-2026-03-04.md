# Bug Fixes & Enhancements (2026-03-04)

Three issues were resolved in this session:
1. Guest table search was throwing a SQL error.
2. QR scanner UI was upgraded with a modern, improved design and better UX.
3. Room images were stuck in an infinite loading loop on the edit page.

---

## 1. Guest Table Search Error

### Problem
Searching in the Guests table produced a database error. The root cause was a typo:

```php
// Before (broken) — 're' is not a real database column
TextColumn::make('re')->searchable()
```

Filament generates a `LIKE` query against every `searchable()` column. Because `re`
does not exist in the `guests` table, this produced an `Unknown column 're' in 'where clause'`
SQL error every time the search field was used.

### Fix
Corrected the column name to match the actual `region` column in the `guests` table:

```php
// After (fixed)
TextColumn::make('region')
    ->searchable()
    ->toggleable(isToggledHiddenByDefault: true),
```

### File Changed
- `app/Filament/Resources/Guests/Tables/GuestsTable.php`

## 2. QR Scanner UI Overhaul

### Problem
The QR scanner modal used the bare html5-qrcode library defaults — a plain white
box with unstyled buttons — which looked inconsistent with the rest of the admin
panel and gave no feedback on the current scanning state.

### Changes

#### `resources/views/vendor/filament-qrcode-field/components/qrcode-input.blade.php`
Complete redesign of the vendored blade component:

| Feature | Description |
|---|---|
| **Dark card** | Glassmorphism-style dark card with indigo gradient border and box shadow |
| **Header** | QR icon, title, subtitle, and a live status badge (Idle → Live → Found) |
| **Scan line** | Animated horizontal indigo beam that sweeps the viewport while scanning |
| **Corner brackets** | Four corner accents mimicking a real camera viewfinder |
| **Success overlay** | Green checkmark + "Booking found! Redirecting…" fades over the viewport on successful decode |
| **Library override** | html5-qrcode's default buttons, selects, and status text are overridden via scoped CSS to match the design |
| **Scanner config** | Added `showTorchButtonIfSupported` and `showZoomSliderIfSupported` for better usability on mobile |

#### `app/Filament/Resources/Bookings/Pages/ListBookings.php`
Improved the Filament `Action` that opens the scanner modal:

| Setting | Before | After |
|---|---|---|
| `modalHeading` | *(default)* | `'Scan Booking QR Code'` |
| `modalDescription` | *(none)* | Descriptive instruction text |
| `modalWidth` | *(default)* | `'md'` |
| `modalCancelActionLabel` | `'Cancel'` | `'Close'` |
| Field label | `'Scan booking QR'` | `''` (blank — the card header replaces it) |
| Error body | *(none)* | `'The scanned QR code did not match any booking. Please try again.'` |

### Files Changed
- `resources/views/vendor/filament-qrcode-field/components/qrcode-input.blade.php`
- `app/Filament/Resources/Bookings/Pages/ListBookings.php`

### Scanner Flow (unchanged)
1. Staff clicks **Scan QR** on the Bookings list page.
2. Modal opens — camera starts automatically.
3. QR is decoded → JSON payload is parsed for `reference` key.
4. Booking is looked up by `reference_number`.
5. On success → redirected to the booking view page.
6. On failure → danger notification with descriptive body text.

## 3. Room Image Infinite Loading on Edit

### Problem
When editing an existing room, the **Featured Image** and **Gallery Images** file
upload fields would spin indefinitely without ever showing the already-saved images.

**Root causes:**

1. **Missing `->disk('public')`** — `SpatieMediaLibraryFileUpload` needs to know
   which disk to read existing media from. Without it, Filament cannot construct
   the preview URLs for already-uploaded images, causing the loader to never
   resolve.

2. **Unconditional `->required()`** on featured image — With `required()` set
   unconditionally, Filament re-evaluates the validation rule every time the
   component hydrates. On an *edit* page the uploader is "empty" from Filament's
   perspective until the existing media is fetched (which was failing due to
   reason #1), trapping the form in a perpetual loading/validation loop.

### Fix

```php
// Before
SpatieMediaLibraryFileUpload::make('featured_image')
    ->collection('featured')
    ->label('Featured Image')
    ->image()
    ->required(),

// After
SpatieMediaLibraryFileUpload::make('featured_image')
    ->collection('featured')
    ->label('Featured Image')
    ->disk('public')                               // ← tells Filament where media lives
    ->image()
    ->imagePreviewHeight('200')                    // ← better UX while uploading
    ->required(fn ($record) => $record === null),  // ← only required on CREATE
```

Same pattern applied to `gallery_images` (minus `required`):

```php
SpatieMediaLibraryFileUpload::make('gallery_images')
    ->collection('gallery')
    ->multiple()
    ->label('Gallery Images')
    ->disk('public')
    ->image()
    ->imagePreviewHeight('150'),
```

### File Changed
- `app/Filament/Resources/Rooms/Schemas/RoomForm.php`

### Notes
- The `public` disk is already the configured default in `config/media-library.php`
  (`MEDIA_DISK=public`), but `SpatieMediaLibraryFileUpload` in Filament v3 requires
  it to be declared explicitly on the component when reading back existing media.
- Image conversions (`card` conversion, 600 px wide) remain registered in
  `Room::registerMediaConversions()` and are unaffected by this change.
- The `required(fn ($record) => $record === null)` pattern is the standard
  Filament v3 approach for fields that are mandatory on create but optional on edit.
